## レビュー観点（優先度順）
1. **security**: SQLインジェクション、XSS、認証漏れ、シークレット露出、CSRF
2. **correctness**: ロジックバグ、エッジケース、null参照、型不整合、off-by-one
3. **data-compat**: スキーマ変更のマイグレーションリスク、API後方互換性、データ形式変更
4. **error-handling**: 例外処理漏れ、retry戦略、graceful degradation、リソースリーク
5. **concurrency**: async/await の不整合、race condition、デッドロック、shared state mutation
6. **perf**: N+1クエリ、不要な再計算、メモリリーク、O(n²) ループ
7. **dependency**: 新パッケージの妥当性、破壊的バージョン変更、未使用依存、ライセンスリスク
8. **maintainability**: 可読性、命名、責務分離、DRY原則、複雑度
9. **testing**: テスト漏れ、カバレッジ不足、テストの信頼性
10. **style**: フォーマット、一貫性（実質的な問題のみ）

## severity 判定基準
- **blocking**: マージ前に修正必須。以下のいずれかに該当:
  - 本番障害を引き起こす（例: 未処理の null で crash、無限ループ）
  - セキュリティ脆弱性（例: SQL injection、XSS、認証バイパス）
  - データ損失・破損リスク（例: マイグレーション漏れ、write without transaction）
  - 正確性バグ（例: 条件分岐の反転、off-by-one で範囲外アクセス）
- **advisory**: 修正推奨だがマージ可能。以下のいずれかに該当:
  - パフォーマンス改善（本番影響が限定的なもの）
  - 可読性・保守性の向上
  - テスト追加推奨
  - ベストプラクティスからの軽微な逸脱

## confidence 判定基準
- **high**: 確実に問題がある（コードの動作から論理的に証明可能）
- **medium**: 高い確率で問題がある（文脈やパターンから推定）
- **low**: 問題の可能性がある（追加調査が必要、または好みの問題に近い）

## 注意事項

- 各issueに具体的な修正案（recommendation）を含めてください
- issueが無い場合は空配列を返してください
- プロジェクト固有ルールやPR/チケット情報が提供されている場合、それを考慮してレビューしてください
